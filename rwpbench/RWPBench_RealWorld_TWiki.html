The rwpbench Tool

This is a description of the rwpbench tool. This tool grew out of the oraunloader program used to unload data from Oracle tables, into a versatile tool that can still fill that job, while also offering a customizable tool for POCs. The following sections describe how to use the tool, then there are sections giving several examples. For unloading data, please get the oraunloader tool instead of this one, as it has more options for that purpose.

The program will record the performance of each SQL statement it executes, either in the database or in files, or send them to a remote system, via a TCP/IP socket. The data contains at least the query name, the execution time, and the returned row count. These can then be used to assess the performance of each query individually, and evaluate the results of the benchmark. The program also offers several options, like taking AWR snapshots at the start/end of or during the run, and printing the awr reports. It can also export the active sqlmonitor for the queries, print execution plans, pga and session statistics.

rwpbench will take it's 'scenario' from the parfile passed as a parameter on the command line. The main process itself never executes any SQL statement included in the POC, it uses forked processes to do so. The awr snapshots and printing awr reports is done by the main process. The forked processes will also do all the specified tasks, like printing session stats, execution plan, sqlmonitor, ... When a process finishes executing a task, it will wait for the next one, until it receives an 'exit' command, then it will just exit. The concurrent number of such processes is governed by a command line parameter' nbr_of_processes' (in previous version, this was specified in the parfile, as the DEGREE parameter). So you'll have at most nbr_of_processes+1 (for the main process) of concurrently running processes. Each process will open a new connection to the database and close it at the end.

rwpbench will fork all the processes it asked to use at the start, and then it will distribute the work on them on first-free/first-served basis. These processes can be on different machines. They connect to the database at the first SQL that arrives, and disconnect at the end.

The section on the command line parameters explains how to start these processes and how they communicate.

Configuration file

The program is executed via command line:

rwpbench parfile=name_of_parameter_file userid=username/password@service master_ip=ip_of_the_master_system master_port=port_number_to_use number_of_machines=how_many_machine_will_run_rwpbench nbr_of_processes=how_many_user_you_want_to_simulate

When running on a single machine, you do not need to specify the ip, port_number or number_of_machines. Note that number_of_machines does not include the current one, so if you have 3 machines in all, you need to specify 2 for this parameter. Here are some examples of how to start the program:

on a single machine: rwpbench parfile=nt.par userid=cq/cq@dbm-taf nbr_of_processes=5 This will start a run, with the parfile called 'nt.par', using db alias "dbm-taf", and user "cq/cq". Five users will be simulated, meaning that 5 processes will be forked on the machine, and you'll have 5 working sessions on the database.

With 3 machines, on the master you start rwpbench like this: rwpbench parfile=nt.par userid=cq/cq@dbm-taf nbr_of_processes=5 master_ip=10.245.13.125 master_port=2049 number_of_machines=2

and then, on each of the 2 other slave machines: rwpbench master_ip=10.245.13.125 master_port=2049

What will happen here is that the master system will wait for the 2 other machines to connect to it, opening a socket using the specified ip/port. It will then ask each system to start a number of child processes depending on the nbr_of_processes argument. Once these child processes are started, the main rwpbench on these 2 machines will exit. The forked processes again open a socket to the main program, which will be used to communicate the tasks to do, as well as control messages. Once all the processes are present, the main instance of rwpbench will start reading the parfile, and distribute the work on the child processes on first-idle/first-served basis. When no more work is available, the main process will wait for all the forked ones to signal that they are idle, and then will ask them to quit, before finishing the final tasks, like taking a snapshot, printing the awr reports, ....

The program will take it's scenario from the parfile. What you can put there is described below. The userid will be used to connect to the database. If you do not specify a 'service', then the current ORACLE_SID will be used.

The output data, if it is printed to a file, will be in a CSV format, where the separator is a ';'.

The parameter file can have any of the following parameters, some of them can be specified several times, each time applying from there on. The parameter names are followed by an '=' sign, with no blanks neither before, nor after it. The names of the parameters are case-insensitive. A main point is that the parfile is processed sequentially. rwpbench will read it line by line, and a task is processed when a 'sql=' line is found. This means that all the parameters specific to a given sql, e.g. 'bind file', 'file', 'log file', must precede the corresponding 'sql' line in the file:

  • query name: allows you to assign a name to the query following this line. This is used to track performance of each query, as the query_name is included in the table that is used to register execution statistics. If the query comes from a file, rwpbench will add the starting line number in the file to the query name you put here. You can also specify 'query name=qb_name'. In this case, and if the queries come from a file, rwpbench will check if there's a hint in the sql of the format 'qb_name(xxxx)'. If it finds one, the string between the '(' and ')' will be taken as the query name and replaces 'qb_name....'. Q_NAME is a supported Oracle hint that is used to identify any SQL or sub-query/in-line view in any statement.
  • print plan stats: if Y/y, the program will print to a log file statistics from V$SQL as well as the execution plan (obtained with dbms_sql.display_cursor). A log file must be specified, otherwise this parameter has no effect.
  • save mystats: if Y/y, rwpbench will save, in a table called my_stats, the rows from v$mystats that changed between during the execution of the statement.
  • sqlmonitor: Use this entry to specify if you want a sqlmonitor report printed for the following query(ies). If you don't want any report, put there 'N'. Otherwise, it can be of the form '[string:]number'. If you specify a string, then it should be the type of report you want. The default is 'ACTIVE'. If the elapsed time of the query is >'number', then the sqlmonitor for this query will be printed. The filename is composed by the query_name_trial_nbr.html. If the 'log file' entry was specified, and it had a directory component, then the sqlmon file will be in that directory, otherwise, it is in your 'current directory'.
  • think time min/think time max: wait time range before starting a given SQL, in seconds. The actual wait time is randomly generated between the 2 limits, using a uniform distribution. Although the values are in seconds, they are actually converted to microseonds, and we use 'usleep' instead of sleep, thus you can specify lower than a second limits (e.g. .1 for _min and .2 for _max to get a think time between 100 and 200 milliseconds)
  • flush bc: if Y/y do an 'alter system flush buffer_cache' before continuing
  • flush sp: if Y/y do an 'alter system flush shared_pool' before before continuing
  • debug file name: if you enable debugging, each process will write it's debug messages to the specified file,after adding it'spid and trial_nbr as suffixes
  • print my stats: if Y:y then the process will print, in the log file, the non-zero values from v$mystat. This is done before exiting the session.
  • print pga stats: ifY/y, the process will print the PGA data from v$process to the log file. This is done after executing any SQL passed to it.
  • trial nbr: Helps identify the run, as it will be also in the execution statistics table. if = N/n then the max from that table is taken and incremented
  • file: The name of the output file to be used for the SQL that comes after. The file name can contain a %d that will be replaced by a serial number
  • log file: the name of the log file, can contain a %d that will be replaced by the process id
  • rows to fetch: maximum number of rows to fetch, if not specified all rows, same will be done if 0 (zero) is specified.
  • max time: If >0 then the queries following this line will be executed several times, until the indicated value, in seconds, expires.
  • exec count: tells the program to execute every sql the specified number of times. if 'max time' is specified, then the 'exec count' is ignored.
  • ramp up time: the time, in seconds, to start the work. It defaults to zero, and is ignored if NBR_OF_PROCESSES<2. It indicates a ramp up period in case you're doing a POC. It is usually useless unless either 'max time' or 'exec count' are specified, or in case you have lots of queries to be taken from a table or a file, so as the processes don't become idle before the period is elapsed. The main program will start a task every 'ramp up time'/NBR_OF_PROCESSES seconds until this ramp up period is finished
  • awr snapshots: Y/y/X will take awr snapshots at before and after the load is processed. If this parameter is a number X, it is means that rwpbench will take a snapshot every X seconds.
  • awr reports: indicates if you want to print awr reports for the snapshots taken when 'awr snapshots' is set to 'Y/y'. This can have the following values: N means no reports, A means to print an awr report for all instances, plus global report; x,y,... comma-sepatated list of instance ids for which you want the reports to be printed. X should be an instance id, or can be 'g'/'G' to get a global report. Reports will be printed to the current directory, and the names are awr_report_[global|instance_id]_begin-snap_end-snap_trial-nbr.html
  • sql: This entry is used to specify the SQL statement to be executed. You have three options here:
    • Either you put a SELECT statement, which can be on several lines, the last line must contain either the following 2 words 'end sql', ';' or '/'. The statement must begin with a 'SELECT' or a 'WITH'.
    • Either you point to an external file like this 'sql=file=[xxx:]name_of_the_file'. So the forked process will read the file indicated to get the SQL statement. Specify the full file name if it is not in the current directory.The file can contain several sql statements, delimited by a ';' or '/'. The '/' must be on a line by itself, while the ';' can also be the last character on the last line of the SQL. If there are several statements, just one output file and log file will be generated, not one for each statement. If you specify the 'xxx:' part, xxx must be a number, and it will tell the system to assign that many processes to execute ALL the SQL statements from the file.
    • Either you point to a table like this 'sql=table=[xxx:]table_name'. So the program will read the SQL statement from table 'table_name'. This table must have at a column called 'sql_text CLOB'. SQL_TEXT will contain the sql statement. The 'xxx:' part is optional as in the 'sql=file' case. The difference with the file case described above is that the statement are read with 'ORDER BY ROWID'.
  • setup script: What you put here is taken to be a SQL or PL/SQL statement that will be passed to all the forked processes to execute just after connecting to the database. It is usually used to to set some session NLS parameters, or PX related settings. It can also be used to do any preprocessing at the start of the benchmark. It cannot be a simple SELECT, but if it is a PL/SQL, it can contains whatever PL/SQL supports. The code can be on multiple lines, and should end with a line containing just 'end setup', or '/' or ';'. It cannot be a file, and the maximum length if 1000 bytes.
  • sqlset: if specified, it should point to an existing sql tuning set, and all the executed statements will be added to this STS.
  • use dbms_sqltune0: if set to Y/y, then rwpbench will use dbms_sqltune0 package to get the sql_id of the statement, otherwise it uses it's ownC code. This can be useful, if for any reason, theC code is wrong. Please note that this code was not tested on anything but an x86 machine, 64 bits.
  • bind file: gives the name of a file containing bind values for the sql statement that follows it. The format of the bind file is described below
  • bind file separator: a character that will be used as a separator in the bind file, defaults to '|'
  • bind results send to host: the name of a host to which to send the results (trial_nbr, query_name, process_id, execution counter, elapsed_time,start_time, returned rows, bind_file_line_nbr) after the execution of a query for each set of bind values. This host must be visible on the network, and you can use the ip address instead of the actual name. For queries not using bind variables, the system will send the same data, but you'll usually have much less rows than when using binds. Attached to this entry are programs written in Python (*.py) that you can use at the receiving end. A note below describes how to use them.
  • bind results send to port: port number to use
  • bind results to db: by default, the statistics of executing queries with bind values are not inserted into the database, unless you set this parameter to Y/y
  • flush results: after how many inserts into response_times the process will commit
  • exec stats file: name of the file where the statistics of executing queries with bind variables will be stored.
  • flush results: Specify a number X. It tells the system to write the statistics of executing with bind values to the file after X executions.
  • stop filename: each forked process, as well as the main routine will check if the file specified here exists, and if it finds it, it will exit gracefully. The checks are done after the execution of any statement, and while fetching rows.
  • bind results to db username: lets you specify a different schema and/or database to use for storing theexecution statistics tables. It should have the following format: username/pwd[@dbalias]
  • check for quit every: this is used in the rows fetching loop. If not set, the loop will not check to see if we want to stop the POC. If it is set to X, then we will chekc for this every X rows. This is meant to reduce the effect of this check on query execution times.
  • rows to spool: gives the number of rows to spool, in case the 'file' parameter is specified. Default is to spool 2<<32-1 rows at most. We may ftech more rows than we spool, depending on 'rows to fetch'.
  • user name: allows you to change the user name when connecting to the database. This take effect for the rest of the file, or until you change it again. If no '
    bind results to db username' is specified then the one you specify on this line will be used to insert data into the 'reponse_times' and 'runs_desc' tables. Give the 'user name' the rights to do so if necessary.

It is to be noted that for each SQL executed, an output file containing the output is created using the name specified by the 'file=' parameter. Each process will 'tag' the file name so that no overwrites occur. The tag is the pid of the process. If no filename is specified, then the data will be just fetched, not printed. A log file will also be generated if you specify a 'log file' entry. The sqlmonitor output will have the same name as the log file, plus a '.html' extension.

The statements in sql files can be DMLs or PL/SQL. They must follow the same 'syntax' as sql*plus scripts: PL/SQL blocks must end with a '/' on a line by itself, while other statements can end with a '/' on a line by itself, or by putting a ';' as the end of the last line. PL/SQL blocks are those that begin with either a 'begin' or a 'declare' words.

The columns from this table mostly come from GV$SQL_MONITOR, except for:

In a benchmark situation, the response_times table can be used to compute and compare execution statistics for each run, which is much easier than having to scan log files. Beware that the query_name column will contain what you put in the 'query name' entry in the parameter file before the SQL=... line. If you use sql=file or sql=table parameters, the starting line number of the SQL in the file, or it's rowid in the table, will be added after this 'query name'. You need to cater for that when retrieving performance data from the table.If the 'query name' parameter is 'qb_name', then you can specify a name for the query in the form on a hint in the SQL itself, as follows: 

select /* qb_name(xxxx)...*/

What you put as 'xxxx' in this 'hint' will be used as a query name in the response_times table.

rwpbench allows you to specify a debug flag that will tell the processes to print tracing messages to the screen, or to a debug file (if the 'debug file name' parameter is specified). As the name implies, these are for debugging purposes only. You should not enable it in normal benchmark, as the output can be very lengthy.

At then end of the run, the program will print a table with the results for each query.

If you want to compile the program, the command is:

make -B -f proc.mk rwpbench

To stop the program prematurely, all you need is to create a file with the name specified in the parameter file. After executing a SQL, the process checks for this file, and stops if it is there. Remember to remove the stop file before restarting the program.

Required Privileges

rwpbench will connect to the database for several purposes:

  • to execute the SQL you submit
  • to insert data into the runs_desc and response_times tables
  • to do some administrative jobs: take awr snapshots, flush buffer cache or shared pool, ...

You can specify the database user for each of these parts.

The administrative jobs will always executed by the user you specify on the command line 'userid=...', so be sure that it has admin privileges (i.e. be able to do 'alter system' to flush buffer cache, .. and also that it can execute dbms_workload_repository functions and procedures.

The user that executes the SQL statements can be specified either by the command line parameter 'userid' or by the parfile parameter 'user name'. This user must have the required privileges to execute the SQL in question. It will try also to access the v$session, v$open_cursors, v$sql, v$sql_monitor and gv$sqlmonitor views. If it cannot, then sql execution statistics, as well as sql_id, ... will be missing from response_times.If you ask for pga stats, then you need to give this user privileges to select from v$process also. For printing session statistics,the user will also need select priviliges on v$mystat and v$statname views. Saving the queries to a sqlsetis done using the DBMS_SQLTUNE package, so the user should be able to execute this package if you are using a sqlset. Printing the sqlmonitor report is also done by calling DBMS_SQLTUNE.report_sql_monitor. It is also advisable to give the user execute privileges on the dbms_sqltune_util0 package. This is used to compute the SQL_ID of the SQL statement. If the privilege is NOT granted, rwpbench will fall back on getting the data from v$open_cursor and v$sql. It will fetch the most recent SQL (using v$sql.last_active_time), having the same first line as the statement being executed, and present in v$open_cursor. If more than one sql fits the bill, then the sql_id may be wrong. This can happen if you have 2 very similar SQL statements executed at less than a second apart. If you encounter this issue, then please give execute privileges on dbms_sqltune_util0 to the user, as this will solve it. If the 'save mystats' is set to 'Y', then the user should also have privileges to select from v$mystats view.

The operations on runs_desc and response_times can be done by a different user than 'userid' by specifying the 'bind results to db username' parameter in the parfile. This user must have the privileges to SELECT/INSERT and UPDATE on these two tables. It does not need any other privileges.

Limitations

There are several structures and variables used in rwpbench that have maximum sizes. These restrict a bit what you can do with the tool. Here are the ones you should know about:

  • Select statements submitted cannot return more than 1000 columns
  • The size of any returned value cannot exceed 500 bytes
  • The name of a column, or it's alias, cannot exceed 300 bytes
  • The output file name, log file name, bind file name and debug file name cannot be more than 512 bytes (including the path)
  • You can use up to 100 machines to execute the benchmark (i.e. number_of_machines<=100)
  • You can have up to 5000 different processes (i.e. nbr_of_processes<=5000)
  • The maximum size of any sql statement, when read from a file or from a table, is 256k bytes. When the statement is in the parfile, the limit is much lower: it cannot exceed 900 bytes. So, if your statements are big ones, please put them in a file, and use the 'sql=file...' parameter.
  • The maximum line size for sql statements read from a file is 1024 bytes.

If you need to change any of these limitations, you'll need to modify the source code and recompile. All of these limitations are "#define"d near the top of the source code.

Using the Python routines

As stated above, attached to this note are three python programs, server_file_only.py and server_qt_simple.py and server_qt_sipmle.py3. Please note that the twiki software is configured to add the .txt extension to Python files, so remove thse after you download the files.

These can be used as a receiver for the data when sent from rwpbench over the network. This requires that python be installed on the machine where you want to run it. All Linux machine has got this, for Windows machine, you'll have to get it from the python.org. The programs offers several possibilities:

  • server_file_only.py will just print the received data to a file.
  • server_qt_simple.py allows for displaying real_time graphs of the received data. the py3 file is the same as py2, but is for python 3 systems.
  • Both programs require the '-p' parameter to tell them which port to use. This must be the same as the 'bind results send to port' in the parfile of rwpbench.
  • For saving the received data to a file: you use the -o option to specify the output file name. Note that data is appended to this file, so old file content is not lost
  • graph the data: for the 'qt_simple' version, you can specify the number of different graphs you want to display, and optionally the list of queries you want to graph (with the -q option). This requires Python version 2.7 to work, along with the numpy and matplotlib modules. If these modules are not installed, the program will tell you so and continue without graphing the data.

To run the program, use the following command: python server.py [-h] -p port_number [-g number_of_graphs -q list_of_queries -o output_file_name ]

  • -p: you must specify the port the program will be using. It must be the same as in the parfile of rwpbench
  • -o: output file name, used to save the data
  • -g: number of graphs to display. These are actually areas on the screens whereline charts of the execution times will be displayed.Queries will be distributed on these areas so as to have more-or-less the same number in each of them.
  • -q: list of queries to graph. If absent all queries are displayed.

When started, the program will display the ip address it is listening on, so you can put this into the parfile. This is very handy when you have several ip addresses available on a machine, like a db node on an Exadata system, or when you have a machine that is using DHCP.

Graphing the data with SQLDeveloper

If you have SQLDeveloper 4.x, and you have the response_times table that is updated with the results, you can use the attached sql_dev_graph_report.xml to view a graph of the response times in real-time.

To do so, you have to import the xml file in sqldev (in the reports pane, click on the 'user defined reports', then right click and use the 'open report' option). You must also have a connection defined for the database and schema where you have the reponse_times table. After that, you can open the report, choose the connection, and voila. SQLDev allows you to have the report automatically refreshed, so that the graph will be updated regularly.

The graph will automatically pick the last trial_nbr. You can edit the SQL in the report to change that, or to make it plot a subset of the queries.

Loading the executions statistics from files

As indicated above, the rwpbench itself, as well as the receiver program can produce files containing the execution statsitics. You can use the following external table description to load this data into the RESPONSE_TIMES table that is used by rwpbench originally. 
drop  TABLE response_times_ext;
CREATE TABLE response_times_ext
  (trial_nbr      number,
   query_name        CHAR(250),
   pid   number,
 time_st number,
   elapsed_t number,
   row_count   number,
   bind_file_line_nbr number)
ORGANIZATION EXTERNAL
  (TYPE ORACLE_LOADER
   DEFAULT DIRECTORY rwpbench
   ACCESS PARAMETERS
     (RECORDS DELIMITED BY NEWLINE
      FIELDS terminated by ';'
      (trial_nbr      ,
   query_name        ,
   pid   ,execution_time_stamp,
   elapsed_t,
   row_count,
   bind_file_line_nbr)
     )
   LOCATION ('tpython.dat')
  );
You'll of course need to adjust the default directory and the file LOCATION to suit your case.

Use cases

A simple benchmark

We will start with a simple benchmark. We have to run 3 sql statements and get the execution times. We will allow for several runs to be tracked, as we change the configuration between the runs. 
trial nbr=N
file=first_sql.csv
query name=Q1
sql=select ....
end sql
file=second_sql.csv
query name=Q2
sql=select ....
end sql
query name=Q3
file=third_sql.csv
sql=select ....
/
In this example, putting 'trial nbr=N' will instruct the main program, before doing the forks, to do a 
select nvl(max(trial_nbr),0)+1
from response_times
and use the returned value as the new trial nbr to pass to the forked processes. If the table does not exist, it will be created. With this configuration, each time you run the tool, you'll get in the response_times table an entry for each executed statement, with the elapsed time, the returned number of rows, and several statistics taken mainly from the V$SQL/v$SQLMONITOR views.

A more complicated benchmark

In this case, using the same SQL as the previous one, bu we will run each query 10 times, with a think time between 1 and 3 seconds before each run. Here's the configuration file: 
trial nbr=N
think time min=1
think time max=3
exec count=10
file=first_sql.csv
query name=Q1
sql=select ....
end sql
file=second_sql.csv
query name=Q2
sql=select ....
end sql
query name=Q3
file=third_sql.csv
sql=select ....
/
The think time before each execution is between 1 and 3 seconds.

Getting more info on the benchmark

When doing benchmarks, you usually want to get the execution plans, and/or the sqlmonitor of the executed statements. You can use the corresponding entries in the configuration file to get this data, like in the sample below: 
print plan stats=y
sqlmonitor=active
trial nbr=N
think time min=1
think time max=3
exec count=10
file=first_sql.csv
log file=q1_log
query name=Q1
sql=select ....
end sql
file=second_sql.csv
query name=Q2
log file=q2_log
sql=select ....
end sql
With the above configuration, you'll get 2 files, with names like specified in the 'log file=' entries, for each query. The first log file will contain execution statistics, plus the plan as given by a call to dbms_xplan.display_cursor. The second file will have an 'html' extension, and will contain the sqlmonitor of the corresponding query, if it was monitored.

A more complicated benchmark

Let's say you got several files given by the client, each containing one SQL statement. The think time is between 0 and 10 seconds, and you have to run these queries for 30 minutes. All the data returned by the SQL is to be fetched but NOT printed. Here's what the configuration file would look like: 
trial nbr=N
think time min=0
think time max=10
max time=1800
stop filename=/tmp/rwpbench.stop
query name=Q1
sql=file=Q1.sql
query name=Q2
sql=file=Q2.sql
Notice the absence of the 'file=' entries. This will tell the processes to fetch all the rows, without actually writing the results to a file. Each query in the file will run as many times as possible till the total elapsed time is above 1800 seconds (30 minutes). As specified above, if you want to stop the process before this time limit, you can create a file called /tmp/rwpbench.stop, and the processes will stop when they finish executing the current query, without waiting for the 1800 seconds (or any 'exec count', if specified).

A yet more complicated case

Say you have a big set of sql statements to run, and you have them in 5 files, each containing several independent SELECTs. You have to run the benchmark using 20 concurrent sessions. You can do this as follows: 
trial nbr=N
think time min=1
think time max=10
sql=file=4:Q1.sql
sql=file=4:Q2.sql
sql=file=4:Q3.sql
sql=file=4:Q4.sql
sql=file=4:Q5.sql

to have the 20 processes, you specify nbr_of_processes=20 on the command line. Four processes will execute the same sql file, and they will execute the statements until they finish all of them. Note that if some statements take much longer than the others, you may end up with less than 20 processes running. A better configuration in this case would be to have the statements in a table, as the load would be better distributed: when a process finishes a query it will take the next available one, and so may execute more queries than the other processes.

Using DDLs

In the MOBI POC, we had a statement that would run much more efficiently if executed on a subset of the data that contains the unique tuples needed. Here is the initial SQL 

SELECT COUNT(DISTINCT C97251_2) AS ANZ 
FROM TPARPPP 
CONNECT BY NOCYCLE
PRIOR C97251_2=C97251_1
START WITH C97251_1 IN (SELECT DISTINCT C97251_1 FROM TPARPPP WHERE MOD(C97251_1, 317) = 0);

As can be seen, this query needs the distinct values of C97251_2, in the different hierarchies. So, if we can have the SQL run on an intermediate table containing the distinct (C97251_1, C97251_2) tuples, if will have much less trees to traverse. For the POC, we were allowed to do this rewrite, as long as we include the time to build this subset in the final tally. To run this in rwpbench, we use the following file, 3g_dist.sql, to define the 'script' 

drop table tparppp_d
/
create table tparppp_d parallel 12
as select distinct c97251_1, c97251_2
from tparppp
/
create index tparppp_d_i1 on tparppp_d(c97251_1, c97251_2) nologging parallel 12
/
alter index tparppp_d_i1  noparallel
/
alter table tparppp_d  noparallel
/
select count(distinct c97251_2)
from tparppp_d
start with mod(c97251_1,317)=0
connect by nocycle prior c97251_2=c97251_1
/
drop table tparppp_d
/

and the parameter file contains the following snippet: 

query name=3g_distinct
sql=file=1:3g_dist.sql

As just one process will execute the sql statements in the above statements taken from file '3g_dist.sql'. We cannot execute this 'query' using multiple concurrent processes (i.e. sql=file=2:3g_dist.sql), as we will probably have an overlap between the CREATE and DROP statements leading to errors.

Using bind variables

In this case, we have 2 sql statements, each using several bind variables. The values to be used for these binds are in 2 files: bind1.q1, and binds.q2. The queries are of OLTP type, so they return in short time, say around 100 micro seconds. In this case, we cannot afford to overload the database with the burden of inserting the execution statistics into the 'response_times' table, so we will send these results to a remote machine, where a special program will handle them. Here's a paramfile to do that: 
trial nbr=N
think time min=1
think time max=2
bind send results to host=slcg02db01.us.oracle.com
bind send results to port=1540
bind file=binds.q1
query name=q1
sql=select ... from ... where c1=:a and c2=to_date(:b,'dd/mm/yyyy')
/
bind file=binds.q2
query name=q2
sql=select ... 
from A, B
where a.c1=b.c1 
and b.c2 between to_date(:a,'dd/mm/yyyy') and to_date(:b,'dd/mm/yyyy') 
and a.c2 between to_date(:a,'dd/mm/yyyy') and to_date(:b,'dd/mm/yyyy')
/

Each process will execute the corresponding query as many times as there are lines in the binds.qx files (minus 1 actually). For q1, assuming that c1 contains only integers, the binds.q1 file will look like this: 

c1:integer|c2:char
1|01/01/2012
10|02/02/2012
.....
For q2, the 'a' and 'b' variables are both character strings, so the binds.q2 file will look like this: 
a:char|b:char
01/01/2012|02/01/2012
02/01/2012|03/01/2012
The elapsed time as well as the returned rows will be sent to the 'slcg02db01.us.oracle.com' server, port 1540. It is supposed that there's some process listening on this port running on that machine.

Analyzing Results

As described in the first sections, you can use the RESPONSE_TIMES/RUNS_DESC tables to see how each query was executed, ... and especially get performance figures for your system. Here's a sample sql*plus script to just that: 
REM query to analyze response times after an rwpbench run
col query_name for a50
col trial for a45
col start_time for a29
break on trial
prompt Enter the run number, 0 for all, -1 to get the last one only
accept run_nbr
select r.trial_nbr||': '||to_char(bt,'yyyy/mm/dd hh24:mi:ss') ||'-'||to_char(et,'yyyy/mm/dd hh24:mi:ss') trial, query_name,wall_time, round(elapsed_sql,2)  elapsed_sql, execs,px_servers,start_time
from (select r.trial_nbr,
query_name, elapsed_t wall_time, elapsed_time/1000000 elapsed_sql, px_servers_allocated px_servers,executions execs,time_st start_time
from response_times r
where &run_nbr=0 or (&run_nbr=-1 and r.trial_nbr in (select max(trial_nbr) from runs_desc))
) r,
(select min(time_stamp_beg) bt, max(time_stamp_end) et, trial_nbr from runs_desc group by trial_nbr) tss
where r.trial_nbr=tss.trial_nbr
order by r.trial_nbr, query_name;

This script will ask for the trial_nbr, and it will give you the last, if you enter -1, all of the trials if you enter 0, otherwise the one you asked for. The following query gives you how many processes were started, how many queries each one executed, and how long it ran: 

col run_span for a40
col time_stamp_beg for a30
col time_stamp_end for a30
break on trial
prompt Enter the run number, 0 for all, -1 to get the last one only
accept run_nbr
select trial_nbr,pid, TIME_STAMP_BEG,  TIME_STAMP_END,TIME_STAMP_end-TIME_STAMP_beg run_span,
(select count(0) from response_times where trial_nbr=a.trial_nbr and process_id=a.pid and time_st between a.time_stamp_beg-numtodsinterval(1,'second') and time_stamp_end) queries_executed
from runs_desc a
where &run_nbr=0 or (&run_nbr=-1 and a.trial_nbr in (select max(trial_nbr) from runs_desc))
order by 1, 4;
